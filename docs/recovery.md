# Pi recovery runbook

What to do when the Pi appliance is broken. Each section: symptom → diagnose → fix.

For cloud-side recovery (when the Pi is healthy but the cloud isn't), see [frutero-fleet/docs/operations.md](https://github.com/HBarefoot/frutero-fleet/blob/main/docs/operations.md).

---

## SD card failure / total Pi loss

### Symptom
- Pi won't boot
- SD card unreadable
- Can't SSH in

### Diagnose
Take the SD card out, try mounting it on another machine. If unreadable / unmountable → dead card.

### Fix

You'll need: a fresh SD card (≥16GB), a Pi OS Bookworm image, and the most recent DB backup downloaded via `/api/security/backup`.

1. **Flash a fresh Pi OS Bookworm** to the new SD card. Use the Imager's "Customize" option to preconfigure SSH + Wi-Fi + the `admin` user with the same password as before so the systemd service comes up cleanly.
2. **Boot, SSH in, clone the repo:**
   ```sh
   sudo apt update && sudo apt install -y git
   cd ~ && git clone https://github.com/HBarefoot/frutero
   cd frutero
   sudo ./install.sh
   ```
   The installer is idempotent — it'll install Node, system deps, build the frontend, seed an empty DB, install the systemd unit, and start the service.
3. **Stop the service** before restoring the DB:
   ```sh
   sudo systemctl stop mushroom-automation
   ```
4. **Restore the DB.** Download `mushroom-backup-YYYY-MM-DD.db.gz` from your backup location, then:
   ```sh
   gunzip mushroom-backup-YYYY-MM-DD.db.gz
   sqlite3 mushroom-backup-YYYY-MM-DD.db 'PRAGMA integrity_check'   # expect "ok"
   sudo cp mushroom-backup-YYYY-MM-DD.db /home/admin/frutero/backend/mushroom.db
   sudo chown admin:admin /home/admin/frutero/backend/mushroom.db
   ```
5. **Restart and verify:**
   ```sh
   sudo systemctl start mushroom-automation
   sudo systemctl status mushroom-automation    # expect "active (running)"
   curl -k https://localhost:3443/api/status     # confirm sensor + actuators
   ```
6. **Log in via the web UI** at `https://<pi-host>.local:3443`. Original users + species + schedules should all be back.
7. **Re-enroll fleet** if the original Pi was paired to the cloud. Cloud's chamber `jti` was tied to the old install. Owner UI on the Pi has a "Pair with cloud" flow that mints a fresh enrollment code on the cloud side.

### What survives, what doesn't

- ✅ Users, schedules, species, alert config, AI advisor history, batches, audit log → all in the DB
- ✅ GPIO pin assignments, relay polarity → all in the DB
- ✅ Cert (auto-regenerated by `install.sh`)
- ❌ Snapshot images / timelapses in `backend/data/` are NOT in the DB backup (per `.gitignore`). If they matter, copy `backend/data/` separately before SD failure (or rsync to a NAS regularly).
- ❌ Cloud fleet pairing — re-enroll via owner UI

## /data corruption / DB won't open

### Symptom
- Service fails to start with `SqliteError: database disk image is malformed`
- Or: WAL/SHM files way larger than the DB itself
- `sqlite3 mushroom.db 'PRAGMA integrity_check'` returns errors

### Diagnose
```sh
sqlite3 /home/admin/frutero/backend/mushroom.db 'PRAGMA integrity_check;'
```
- `ok` → DB is fine; problem is elsewhere (env, permissions, port).
- Anything else → corruption. Note what it says.

### Fix

**Try recovery first:**
```sh
sudo systemctl stop mushroom-automation
cd /home/admin/frutero/backend
cp mushroom.db mushroom.db.broken
sqlite3 mushroom.db.broken '.recover' | sqlite3 mushroom.db.recovered
sqlite3 mushroom.db.recovered 'PRAGMA integrity_check'  # hopefully "ok"
mv mushroom.db.recovered mushroom.db
sudo systemctl start mushroom-automation
```

**If `.recover` doesn't yield a clean DB, restore from backup:**
```sh
sudo systemctl stop mushroom-automation
cd /home/admin/frutero/backend
mv mushroom.db mushroom.db.broken
gunzip -c ~/backups/mushroom-2026-04-XX.db.gz > mushroom.db
sudo chown admin:admin mushroom.db
sudo systemctl start mushroom-automation
```

You'll lose any data created between the backup time and the corruption point. Audit log + readings + activity feed since the backup are gone. Schedules + actuators + species are usually static so they typically aren't lost.

**If WAL is huge but DB is fine:**
```sh
sudo systemctl stop mushroom-automation
sqlite3 mushroom.db 'PRAGMA wal_checkpoint(TRUNCATE);'
sudo systemctl start mushroom-automation
```
Forces the WAL to flush + truncate. Usually means a previous unclean shutdown left it dirty.

## Mid-photoperiod restart leaves lights off

### Symptom
- Restarted the service mid-day
- Lights should be on (it's daytime per the schedule) but they're off

### Diagnose
This was a real bug pre-Phase-7b. Since `d7f94cf` (boot-time GPIO state recovery, see `project_mushroom_automation.md`), `gpio.restoreScheduledStates()` runs at boot and re-applies the most-recent past schedule action for latching actuators.

Check journal:
```sh
journalctl -u mushroom-automation --since "10 minutes ago" | grep -E "boot-restored|startup_recovery"
```
Should see `[gpio] boot-restored light=ON from schedule` if the recovery ran.

### Fix
- If the line is there, the recovery ran — check whether the GPIO was actually pulsed (relay click + visible light state).
- If the line isn't there, the schedule is malformed or the actuator's `auto_off_seconds` is set (recovery skips those — they're treated as transient pulses, not latched states). Check `/api/actuators/<key>` for the schedule + auto-off config.

Manual override always wins:
```sh
curl -k -X POST -H 'Content-Type: application/json' \
  -d '{"on":true}' \
  -b session.cookie \
  https://localhost:3443/api/devices/light
```

## DHT22 sensor silence / stale readings

### Symptom
- Dashboard banner: "Sensor silent for X minutes"
- `/api/status.sensor_health` shows `silent: true`
- Mister automation isn't firing (correctly — automations.onSensorReading short-circuits when silent to prevent stale-humidity dry-fires)

### Diagnose
1. **Check journal for sensor errors:**
   ```sh
   journalctl -u mushroom-automation --since "1 hour ago" | grep -E "sensor|dht|DHT"
   ```
   Common: `sensor read failed: timeout` (loose wire), `EBUSY` (gpiod conflict), `Permission denied` (admin user not in gpio group).
2. **Check wiring physically.** DHT22 on GPIO 4. Three wires: VCC (3.3V), GND, DATA. The DATA line needs a 10kΩ pull-up to VCC — if the breadboard joint is loose, you get sporadic timeouts.
3. **Check the kernel module hasn't conflicted:** `lsmod | grep w1` — if `w1-gpio` is loaded against GPIO 4, it conflicts with DHT. Should not be loaded by default per `install.sh`.

### Fix

**Hardware-side:**
- Reseat all three wires + the pull-up resistor.
- Power-cycle the Pi (the sensor recovers fine; the watchdog will clear once readings resume).

**Software-side restart:**
```sh
sudo systemctl restart mushroom-automation
```
Watchdog re-arms after `[sensor] start polling` log line.

**If it persists after reseat + restart**, the DHT22 itself may have failed — they're cheap, swap it. Order a DHT22 (not DHT11) and rewire identically.

## Cert expired / browser warns "not secure"

### Symptom
- Browser blocks `https://<pi>.local:3443` with "certificate has expired"
- Pi logs OK; web UI just unreachable

### Diagnose
```sh
sudo openssl x509 -in /etc/ssl/frutero/cert.pem -noout -dates
```
If `notAfter` is in the past → expired.

### Fix
```sh
cd /home/admin/frutero
sudo ./rotate-cert.sh
```
Generates a fresh ECDSA P-256 cert (10y validity) with SANs covering `<host>.local`, LAN IP, Tailscale, Cloudflare. Restarts the service. Browser will need to re-trust the new cert (delete old exception, re-accept new).

## Mister atomizer dry-fires suspected

### Symptom
- Mister fires when humidity is already at threshold
- Or: relay click but no visible mist
- Or: rapid mister cycling (potential disc damage)

### Diagnose
1. **Check the actuator's safety status:**
   ```sh
   curl -k -b session.cookie https://localhost:3443/api/devices/mister
   ```
   Returns `{state, safety_status: {min_off_remaining_seconds, daily_used_seconds, daily_remaining_seconds}}`. If `daily_remaining_seconds` is 0, the daily clamp blocked further fires — that's the safety doing its job.
2. **Check the actuator config:**
   ```sh
   curl -k -b session.cookie https://localhost:3443/api/actuators/mister
   ```
   Should show `config.safety = {max_on_seconds: 30, min_off_seconds: 30, daily_max_seconds: 1800}` and `auto_off_seconds: 10`. If any of these are missing or 0, the disc is unprotected.
3. **Check Activity feed in the UI** for the trigger column. Three trigger paths exist (api / automation / schedule) with different auto-off behavior — see `project_mister_runtime_behavior.md` in memory.

### Fix
- If safety config is missing, **edit the actuator** via the Hardware page or `PUT /api/actuators/mister` to add the safety block. Future actuator updates should also enforce safety on update (currently only on create).
- If hardware is fine but the disc seems to dry-fire, the wick may have detached or the water tank emptied. Check water level + wick contact.

## fleet-agent disconnected from cloud

### Symptom
- `/api/fleet/status` shows `connected: false` or `last_error: 'revoked_by_cloud'`
- Cloud's `/admin/customers/:id` shows the chamber's `last_chamber_seen` is stale

### Diagnose
```sh
curl -k -b session.cookie https://localhost:3443/api/fleet/status
```

Common values for `last_error`:
- `'revoked_by_cloud'` → cloud archived this chamber or the org was suspended. Check cloud-side first.
- `'org_suspended'` → cloud's org has `suspended_at`. Operator must unsuspend.
- `null` but no recent `last_heartbeat_at` → network issue between Pi and cloud. Check internet connectivity.

### Fix

**Re-enroll if revoked:**
1. Cloud-side: chamber must be unarchived OR a fresh enrollment code minted.
2. Pi-side: open the Security page → "Pair with cloud" → paste the new enrollment code. fleet-agent picks it up and resumes heartbeats within ~60s.

**Network issue:**
- `ping cloud.your-domain.com` from the Pi
- Check Pi's outbound HTTPS isn't blocked by router firewall (rare on residential networks)
- Tailscale users: confirm Tailscale is up (`tailscale status`)

## Hardware swap mid-grow

### Symptom
- Need to replace a relay module, the Pi itself, or move chambers
- Active grow batch in progress (don't want to lose the timeline / scheduled actions)

### Diagnose
What's the smallest change that achieves the goal?

### Fix — relay module swap
1. **Stop the service** so GPIO writes don't fire mid-swap:
   ```sh
   sudo systemctl stop mushroom-automation
   ```
2. **Power down** the Pi.
3. **Verify wiring polarity matches** — the new relay must be the same trigger polarity (low-trigger vs high-trigger). If different, you'll need to update `actuator.inverted` in the DB after swap.
4. **Rewire to identical pins** (GPIO 17/18/27 by default).
5. **Power up + start service:**
   ```sh
   sudo systemctl start mushroom-automation
   ```
6. **Test each actuator pulse** via the Hardware page → pulse 200ms each. Confirm relay click + correct downstream behavior.

### Fix — Pi swap (move SD card to new Pi)
1. Stop service, power down old Pi.
2. Move SD card to new Pi. Boot.
3. Hostname stays the same; if ssh fails, regenerate ssh host keys: `sudo ssh-keygen -A`.
4. Cert is bound to hostname so should keep working. If not, `sudo ./rotate-cert.sh`.
5. Service auto-starts, `gpio.restoreScheduledStates()` re-applies any latched outputs (lights mid-photoperiod, etc).

### Fix — chamber swap (move Pi to a new chamber)
- Software state stays. Just move the hardware.
- Update species/batch records via the UI to reflect the new chamber.

## Schema drift between branches

### Symptom
- Pulled a new branch, restarted, service errors with `no such column` or `no such table`

### Diagnose
All schema migrations are additive + idempotent (`ensureColumn`, `CREATE TABLE IF NOT EXISTS`). If a column or table is genuinely missing, the migration should add it on the next boot.

```sh
journalctl -u mushroom-automation --since "5 minutes ago" | grep -E "ensureColumn|migration|schema"
```

### Fix

If the service won't start cleanly, try a clean boot:
```sh
sudo systemctl restart mushroom-automation
```

If it still fails, the schema may need manual addition. Compare against `backend/database.js` `init()` for the missing piece. Worst case, restore from backup.

**Going BACK to an older commit (downgrade):** older commits may not understand newer columns and could write garbage. Only downgrade if absolutely necessary, and back up first.

## Standard responses

### "Is the Pi alive?" diagnostic

```sh
sudo systemctl status mushroom-automation       # service running?
curl -k https://localhost:3443/api/status        # health?
journalctl -u mushroom-automation --since "5 minutes ago" | tail -50   # any errors?
df -h /                                          # disk full?
free -h                                          # OOM?
vcgencmd get_throttled                           # power-supply or thermal throttle?
```

### Escalation order

1. **Read the journal first.** Most "what happened?" questions are answered by `journalctl`.
2. **Check the Activity feed in the UI second.** Most "why did the actuator fire?" questions are visible there.
3. **Check `/api/hardware/scan` third** for hardware-level state (sensor reading freshness, GPIO availability, V4L2).
4. **Open a shell to the DB last.** Direct SQLite access is destructive territory; use only when the previous three didn't surface the issue.
